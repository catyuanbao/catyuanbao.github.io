背景

本文主要记录《Hands-On System Programming with Linux》一书中第一章的内容，主要是Linux系统编程的基本概念，Linux的一些设计哲学，以及一些简单代码。

学习本章，可以掌握一些基本概念，比如printf到底是不是系统调用，这类问题。

Linux 系统哲学

  1. Everything is a process; if it’s not a process, it’s a file
  2. One tool to do one task
  3. Three standard I/O channel
  4. Combine tools seamlessly
  5. Plain text preferred
  6. CLI, not GUI
  7. Modular, designed to be repurposed by others
  8. Provide the mechanism, not the policy

进程or文件（所有玩意儿都是进程，不是进程就是一个文件）

The Unix design philosophy abstracts peripheral devices (such as the keyboard,
monitor, mouse, a sensor, and touchscreen) as files – what it calls device files. 

By doing this, Unix allows the application programmer to conveniently ignore the details
and just treat (peripheral) devices as though they are ordinary disk files.

Linux把设备都抽象为文件，这么做的好处在于开发者可以忽略设备的细节，只按照一般文件使用设备接口即可。

每个进程都有三个fd

  1. Standard input (stdin: fd 0): The keyboard device, by default
  2. Standard output (stdout: fd 1): The monitor (or terminal) device, by default
  3. Standard error (stderr: fd 2): The monitor (or terminal) device, by default

一个组合命令：

$ sort < fruit.txt > sorted.fruit.txt 2> /dev/null

这个命令作用就是从fruit.txt作为sort的输入，排序后将结果输出到sorted.fruit.txt 文件，同时，把标准错误重定向到/dev/null.

几个命令：

df --local > tmp
sed --in-place '1d' tmp
sort -k4nr tmp

第一行命令是列出来所有本地已经挂载的文件系统，一般用于查看磁盘容量。

第二行命令是删除tmp文件第一行，直接在文件上修改。

第三行是排序tmp文件，按照第四个字段排序，并逆序排序。

Three standard I/O channels（三个标准IO channels）

之前已经简单介绍过了，就是标准输入，标准输出，标准错误。

这里简单提一下stdout的行缓冲机制，标准输出（stdout） 一般采用行缓冲机制，特别是在与终端（控制台）交互时。

基本原理：

  • 当程序输出数据到 stdout 时，数据会被先存储在行缓冲区。
  • 当缓冲区遇到换行符（\n）或缓冲区满时，数据会被实际写入到输出设备。
  • 如果缓冲区没有遇到换行符或没有满，数据会暂时保存在缓冲区中，等待进一步的输入。

比如以下Python代码：

import time

print("Hello, world!", end="")  # 使用end=""避免换行符
time.sleep(2)  # 等待2秒
print(" Still here!")  # 由于行缓冲机制，上面的内容和这一行会一起显示

Python提供了内置的-u选项，支持运行脚本/程序无缓冲：

$ python3 -u buffer.py
Hello, world! Still here!

Hello world!会被直接打印出来。

stderr是无缓冲的，因为它是处理程序报错的输出，需要比较及时，所以，设计时没有缓冲机制。

使用管道可以将前一个命令的输出作为后一个命令的输入，这样数据流动下去。

Plain text preferred（首选纯文本）

文本的通用性强，不要搞复杂的东西。

CLI, not GUI（使用命令行，而不是图形界面）

一般管理服务器都是命令行，不安装图形界面。

Modular, designed to be repurposed by others （模块化，可供他人重新利用）

设计软件常用的方法论-复用。

Provide mechanisms, not policies（提供机制，而不是政策）

实例：

Linux 提供了基本的安全机制，如用户和组权限、SELinux、安全模块等。

这些机制提供了保障系统安全的基础工具，但具体如何配置和使用这些工具，取决于管理员的策略。

例如，管理员可以根据需求开启或关闭 SELinux，设置不同的安全策略。

优点在于模块化，可以配置，灵活选择。

LInux 系统架构

这里的系统架构主要讨论3个主题。

  1. The processor Application Binary Interface (ABI).
  2. CPU privilege levels.
  3. How these affect the code we write.

ABI

Linux ABI（Application Binary Interface，应用二进制接口）是指应用程序与操作系统内核之间的接口定义，包括数据类型、系统调用、库函数等。

这些接口定义了应用程序在二进制级别如何与操作系统和其他软件组件交互，确保应用程序能够在不同的系统环境中运行而无需重新编译。

主要由以下几个部分：

  1. 系统调用接口（System Call Interface）
  2. 库接口（Library Interface）
  3. 二进制格式（Binary Format）
  4. 调用约定（Calling Convention）
  5. 数据类型和内存布局（Data Types and Memory Layout）

以一个简单的C代码为例：

#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>

int main(void) {
    int a;
    printf("Hello, Linux System Programming, World!\n");
    a = 5;
    exit(0);
}

Accessing a register’s content via inline assembly(使用内联汇编获取寄存器内容)

通过内联汇编语句，获取到寄存器RCX内容：

#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>
typedef unsigned long u64;

static u64 get_rcx(void) {
    __asm__ __volatile__(
        "push %rcx\n\t"
        "movq $5, %rcx\n\t"
        "movq %rcx, %rax");
    __asm__ __volatile__("pop %rcx");
}

int main(void) {
    printf("Hello, inline assembly:\n [RCX] = 0x%lx\n",
    get_rcx());
    exit(0);
}

运行结果：

$ ./a.out
Hello, inline assembly:
 [RCX] = 0x5

修改代码：

#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>
typedef unsigned long u64;

static u64 get_rcx(void) {
    __asm__ __volatile__("movq %cr0, %rax");
}

int main(void) {
    printf("Hello, inline assembly:\n [RCX] = 0x%lx\n",
    get_rcx());
    exit(0);
}

运行结果：

$ ./a.out
[1]    2463 segmentation fault (core dumped)  ./a.out

为什么出现这种情况？

CPU privilege levels(CPU特权等级)

代码报错的原因与特权级别有关。

在 x86 架构中，控制寄存器（如 cr0）只能在内核模式（Ring 0）下访问，而你的程序在用户模式（Ring 3）下运行，因此无法访问这些寄存器。

那么什么是特权等级，有什么特权等级？

在计算机系统中，CPU 操作权限通常被划分为不同的级别，以确保系统的安全性和稳定性。

最常见的划分是 用户模式（User Mode） 和 内核模式（Kernel Mode），也称为 监督模式（Supervisor Mode） 或 特权模式（Privileged Mode）。

下面是特权模式和用户模式的对比，

因为上面的代码试图访问CR0控制寄存器的内容，这在用户模式中是不允许的，以及仅在Ring 0特权下允许！

只有操作系统或内核代码才能访问控件寄存器。

别名 内核模式 用户模式
特权级别 Ring 0 Ring 3
典型使用者 操作系统内核、设备驱动程序 用户应用程序、标准库
访问权限 完全访问所有硬件资源 受限访问，不能直接访问硬件资源
特权指令 可以执行 不能执行

错误处理 处理器异常、系统崩溃恢复 通常不能处理，需要内核介入

Libraries

我们经常使用的printf就是一个Libraries函数，通常被放在GNU libc (glibc). 库函数下面还有一层，叫做系统调用接口(system all)

system call

系统调用实际上是内核功能，可以通过glibc存根例程。

它们起着关键作用；它们将用户空间连接到内核空间。

如果用户程序想要请求内核的某些内容（从文件读取，写入网络，更改文件的权限），它通过发出系统调用来做到这一点。
因此，系统调用是内核的唯一合法入口点。没有其他可能。

在程序调用到系统调用时，CPU就会陷入到内核态。

以hello.c举例：

#include <stdio.h>
#include <unistd.h>
#include <stdlib.h>

int main(void) {
    int a;
    printf("Hello, Linux System Programming, World!\n");
    a = 5;
    exit(0);
}
